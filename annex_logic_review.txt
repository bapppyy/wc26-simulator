1. DATA STRUCTURE
-----------------
ANNEX_C is a plain JS object with 495 entries — one per possible
combination of 8 groups whose third-placed teams make the best 8.

Key format:  "ABCDEFGH"  (sorted group letters of the 8 qualifying groups)
Value format: { "A":"H", "B":"G", "D":"B", "E":"C", ... }

  Inner key   = group whose WINNER faces a third-placed team in R32
  Inner value = group whose THIRD-PLACED team is slotted as that opponent

Only 8 groups produce bracket slots that host third-placed teams:
  A, B, D, E, G, I, K, L  (groups C, F, H, J always face 2nd-place teams)

3. BOSNIA vs. TURKEY QUESTION
------------------------------
User concern: "Bosnia appears disproportionately against D1 (Turkey)."

TEAM GROUP ASSIGNMENTS (from data.js):
  Bosnia-Herz. → Group B
  Turkey       → Group D

HOW THIS PAIRING ARISES:
  - Turkey is 1D (Group D winner)
  - Bosnia-Herz., when qualifying as a best third-placed team, is "3B"
    (the third-placed team of Group B)
  - engine.js M6: [g('D', 0), t3(annexC['D'])]
    → 1D plays the third-placed team from the group specified by annexC['D']

FREQUENCY ANALYSIS (when D qualifies in top-8 third-place groups):
  Group D winner (Turkey) faces:
    3B  — 210 / 330 scenarios = 63.6%  ← Bosnia-Herz. if they finish 3rd
    3J  —  45 / 330 scenarios = 13.6%
    3E  —  33 / 330 scenarios = 10.0%
    3I  —  31 / 330 scenarios =  9.4%
    3F  —  11 / 330 scenarios =  3.3%

  3D (Bosnia-Herz. group's third-placer) faces:
  NOTE: Bosnia is Group B, not D — but what about the 3rd placer from Group B?
    1E  — 212 / 330 scenarios = 64.2%
    1I  —  99 / 330 scenarios = 30.0%
    1K  —  19 / 330 scenarios =  5.8%

CONCLUSION FOR TURKEY vs. BOSNIA FREQUENCY:
  1D (Turkey) faces 3B (which could be Bosnia-Herz. if they finish 3rd in B)
  in 63.6% of scenarios where Group D qualifies a third-placed team.

  This is NOT a bug. It reflects the actual FIFA Annex C table where the
  slot assigned to face 1D is predominantly filled by the 3rd-place team
  from Group B. This is the official FIFA design — it ensures geographic
  and group-origin separation while managing bracket structure.

  Why does 3B face 1D so often (63.6%)? By design, in the vast majority
  of 8-group combinations that include D, the Annex C table routes 3B to
  face 1D. This is the official table — it is not an error.

--------------------------------------

  M0:  1E vs 3(annexC['E'])
  M1:  1I vs 3(annexC['I'])
  M2:  2A vs 2B          ← pure 2nd-place matchup
  M3:  1F vs 2C          ← 1st vs 2nd
  M4:  2K vs 2L          ← pure 2nd-place matchup
  M5:  1H vs 2J          ← 1st vs 2nd
  M6:  1D vs 3(annexC['D'])
  M7:  1G vs 3(annexC['G'])
  M8:  1C vs 2F          ← 1st vs 2nd
  M9:  2E vs 2I          ← pure 2nd-place matchup
  M10: 2D vs 2G          ← pure 2nd-place matchup
  M11: 1J vs 2H          ← 1st vs 2nd
  M12: 1L vs 3(annexC['L'])
  M13: 1B vs 3(annexC['B'])
  M14: 1K vs 3(annexC['K'])
  M15: 1A vs 3(annexC['A'])

The 8 slots that receive third-placed teams (M0, M1, M6, M7, M12–M15)
correspond to groups A, B, D, E, G, I, K, L — verified to be the only
keys present in every ANNEX_C entry.

POTENTIAL EDGE CASE IDENTIFIED:
  If the top-8 third-placed combination does NOT match any key in ANNEX_C
  (i.e., annexC = undefined), the engine falls back to r32Pairs = [].
  This means ALL 16 R32 matches would be null/bye matches, and no
  third-placed teams would advance. The engine handles null pairs gracefully
  (pushes null to r32Ms/r32W), but the tournament would continue with only
  16 teams from 2nd-place and 1st-place groups — which is incorrect.

  However, since ANNEX_C covers all C(12,8) = 495 combinations, this
  scenario cannot occur in practice. Risk: NONE.
